How the Three Upper-Bound Columns Are Computed

The live table on the main page lists three different upper-bound mechanisms side by side. They are complementary: the first two are algebraic certificates extracted from the APM structure, while the third comes from decoder-fail simulations. The displayed code parameter \(\left[\left[n,k,\le u\right]\right]\) uses the smallest currently available upper bound for that row.

Column What it measures Typical parameter-file fields
Latent UB An upper bound obtained from latent witnesses built from the \(\Psi_3\) kernel. latent_x_upper, latent_z_upper, latent_min_upper
\(m\)-block UB A non-latent upper bound obtained from block-constant compression and lifting. nonlatent_x_upper, nonlatent_z_upper, nonlatent_min_upper
Decoder-fail UB A direct upper bound coming from a syndrome-matched logical residual observed in simulation. decoderlog_x_upper, decoderlog_z_upper

1. Latent Distance Upper Bound

For the \((J,L)=(3,12)\) family considered here, a low-weight latent logical operator can be produced by searching the kernels of \(\Psi_3\) and \(\Psi_3^{\mathsf T}\). A nonzero kernel vector \(\mathbf{u}\) or \(\mathbf{v}\) is converted into a codeword candidate such as \(\mathbf{u}^{\mathsf T}\widetilde H_X\) or \(\mathbf{v}^{\mathsf T}\widetilde H_Z\), and the resulting support size gives a latent upper bound.

In the parameter files this appears as latent_x_upper, latent_z_upper, and latent_min_upper. The live table reports the minimum of the \(X\) and \(Z\) latent witnesses.

2. \(m\)-Block-Constant Non-Latent Upper Bound

The second algebraic bound uses a divisor \(m\mid P\). One compresses the affine-permutation construction to a quotient problem of size \(P/m\), finds a low-weight witness there, and then lifts it back to the full code. In the non-latent case, the lifted vector must remain outside the opposite row space so that it certifies a genuine non-stabilizer logical operator.

This is why the parameter files also record the best block sizes nonlatent_x_m and nonlatent_z_m. The live table shows the resulting non-latent minimum upper bound as the \(m\)-block UB column.

3. Decoder-Fail-Based Upper Bound

The third bound does not come from algebraic compression. Instead, we run decoder-fail campaigns at \(p=0.03\) and search for failures where the decoder output matches the observed syndrome but differs from the true error by a non-stabilizer logical residual. If the residual has weight \(w\), then the corresponding logical distance is at most \(w\).

This is recorded as decoderlog_x_upper or decoderlog_z_upper. If no logical fail has been observed yet, the table shows -- in the decoder-fail column.

The decoder-fail campaigns are used to collect logical-error witnesses, not to optimize the error floor itself. A row can therefore have a very tight algebraic bound even if the decoder-fail column still shows --, and conversely a decoder-fail witness can tighten a row beyond its original structural bound.

How the Live Table Uses Them

The live table keeps all three ingredients visible because they play different roles. For example:

  • A row such as \(P=288\) can have latent and \(m\)-block bounds equal to 24 while the best decoder-fail witness is still 28.
  • A row such as \(P=240\) was tightened by decoder-fail sampling from a structural value of 24 down to 16.
  • The separate \(P=768\) paper-code reference row is an exact transcription of the printed Table 1 parameters in arXiv:2601.08824. For that exact row, the currently tracked algebraic information is an \(m\)-block non-latent upper bound of 32.
